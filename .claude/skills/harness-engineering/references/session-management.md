# セッション間の状態管理

## 問題の構造

エージェント各セッションは前セッションの記憶を持たない。毎回新しい作業員が何も引き継ぎなしで来るのと同じ。

## 起動ルーチン標準化（Anthropicパターン）

各セッション開始時にエージェントが以下を実行：

1. 作業ディレクトリ確認
2. Gitログと進捗ファイル読み取り
3. 機能リストから次の最優先タスク選択
4. 開発サーバー起動と基本機能疎通テスト

このルーチンで前セッションが壊れた状態で終了していても即座に検出・修復。

## Gitをセッション間のブリッジに

- 各セッション終了時に記述的コミットメッセージでGitコミット
- 次セッションの`git log --oneline -20`が最も信頼性高い「何が起きたか」記録
- Gitログはコード変更と一対一紐づくため、記述ドキュメントと異なり構造的に腐敗しにくい

## 進捗記録にJSONを

Anthropic知見：機能リスト・進捗記録はMarkdownよりJSON適。

- モデルはJSON形式データを不適切に編集・上書きする可能性がMarkdownより低い
- 注意：短期プロジェクト向け。長期では機能リスト自体が腐敗リスク
- テストスイートそのものを機能リスト代替として検討すべき

## 計画と実行の分離

> "計画と実行の分離は、私が行う最も重要な単一のことだ" - Boris Tane (Cloudflare)

- エージェントにまず計画を立てさせ、人間がレビュー・承認してから実行
- 大目標を小構成要素に分解、各完了後に次に進む（ワンショット問題の回避）
- 明示的にブラウザ自動化ツール等でのE2Eテスト指示で精度向上

## プラットフォーム比較：Claude Code vs Codex

### Claude Code（作業場型）
- 開発者環境に直接入り、ファイル編集・コマンド実行をローカルで実行
- Hooksシステムで毎回のツール実行に決定論的制御挿入
- Plan Mode + Extended Thinkingで読み取り専用計画モード

### Codex（密室型）
- クラウドサンドボックス（ネットワーク隔離コンテナ）で独立作業
- 複数タスク非同期並列実行可能
- 現時点でPreToolUse/PostToolUse相当のHooksなし

### ハイブリッド戦略（2026年主流）
1. Claude Codeで計画・設計
2. Codexで並列実行
3. Claude Codeでレビュー・改善

## 意思決定フレームワーク

| 最優先事項 | 推奨 | 理由 |
|----------|------|------|
| 品質 | Claude Code主軸 | Hooksによる決定論的品質ゲートは代替手段なし |
| スループット | Codex主軸 | 非同期サンドボックス並列実行は代替手段なし |
| 両方 | ハイブリッド | ハーネスの品質がスケール時の信頼性を決定 |
