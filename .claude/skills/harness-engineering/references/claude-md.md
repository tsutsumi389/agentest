# CLAUDE.md設計ガイド

## 基本原則：ポインタとして設計

CLAUDE.mdは実行可能アーティファクト（テスト、リンター設定、ADR）へのポインタ。説明文書ではない。

## サイズの目安

**理想：50行以下**

- Anthropic公式：「200行以下」は上限であって目標ではない
- IFScale研究：150-200指示時点でprimacy bias（先頭指示への偏り）が顕著
- Claude Codeシステムプロンプト自体が約50指示 → CLAUDE.mdが100行あれば計150指示

## 書くべきもの

- **ルーティング指示** - `pnpm test`で走る、ADRは`/docs/adr/`、アーキテクチャルールは`archgate check`
- **禁止事項一覧** - 各項目がADRまたはリンタールール参照
- **ビルド・テスト・デプロイ最低限コマンド**

## 書くべきでないもの

- **システム現状説明** - コードとテストが真実のソース
- **技術スタック解説** - エージェントはpackage.jsonやgo.mod読める
- **冗長なコーディングスタイルガイド** - リンターとフォーマッターに委ねる

## 実践設計

1. **ルートファイル50行以下** - 最小限の事実、Skills・MCPポインタのみ
2. **詳細はオンデマンドロード** - Skills、`.claude/rules/`、サブディレクトリ別AGENTS.mdで分割
3. **圧縮積極的** - Vercelは40KBを8KBに圧縮でも100%パス率維持
4. **各行に問う** - 「この行を消したら間違えるか？」→ Noなら削除

## ポインタが腐敗した場合

ポインタ設計の副次的利点：指すファイルパスが存在しなくなれば404相当エラー発生。
- 腐敗が機械的に検出可能
- 記述的ドキュメントの腐敗は沈黙のうちに進行
- 壊れたポインタは騒がしく失敗する

## リポジトリ衛生

### 置くべきもの
- 実行可能アーティファクト（コード、テスト、リンター設定、型定義、スキーマ、CI設定）
- ADR（タイムスタンプとステータス明示で構造的に有効性判定可能）

### 置くべきでないもの
- 説明文書「現在のシステムはこうなっている」
- 設計概要や手書きAPI説明
- アーキテクチャ概要図の説明テキスト

理由：必然的に腐敗し、エージェントが古い情報を真実として採用するリスク。

## ADRの重要性

Architecture Decision Records：
- 不変原則により、エージェントがgrepで発見した過去決定も安全
- 置換時にステータスで明示されるため、現在有効な決定を構造的に判別可能
- archgateパターン：各ADRにcompanion `.rules.ts`ファイルを持たせ、決定を実行可能チェックにエンコード
