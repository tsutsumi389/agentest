# 機能一覧

## 概要

テスト管理ツール「Agentest」の機能要件。Coding Agent との連携を前提とした設計。

---

## 非機能要件

### パフォーマンス

| 項目 | 要件 |
|------|------|
| API レスポンス | 95 パーセンタイルで 500ms 以内 |
| リアルタイム反映 | Agent 操作から 1 秒以内に Web 画面に反映 |
| 検索レスポンス | 全文検索は 2 秒以内 |
| 同時接続 | WebSocket 1,000 接続/インスタンス |

### データ保持

| 項目 | 要件 |
|------|------|
| 履歴保持期間 | 無期限（ストレージ容量による制限） |
| 履歴件数上限 | エンティティあたり 1,000 件（超過時は古いものを削除） |
| スナップショット保持 | 実行結果と同期間保持 |
| 論理削除データ | 30 日後に物理削除 |

### 可用性

| 項目 | 要件 |
|------|------|
| 稼働率 | 99.9%（月間ダウンタイム 43 分以内） |
| バックアップ | 日次フルバックアップ、1 時間ごとの差分 |
| リカバリ | RPO 1 時間、RTO 4 時間 |

### Agent セッション

| 項目 | 要件 |
|------|------|
| ハートビート間隔 | 30 秒 |
| ロック有効期限 | ハートビート途絶から 60 秒後に自動解除 |
| セッションタイムアウト | 無操作 30 分で自動終了 |

## 機能詳細

### TC-004: テストケース参照入力

**動作:**
1. テストケースのタイトル入力欄で `@` を入力
2. テストスイート名の候補をリスト表示
3. スイート選択後 `/` を入力
4. テストケース名の候補をリスト表示
5. 選択するとそのテストケースの内容をコピー

**入力例:**
```
@ログイン機能/正常系ログイン
```

### 履歴管理（PRJ-004, TS-004, TC-005）

**対象:**
- プロジェクト
- テストスイート
- テストケース

**記録対象:**
- 変更日時
- 変更者（人 or Agent）
- 変更内容（差分）
- 変更理由（任意）

**機能:**
- 履歴一覧表示
- 差分表示
- 任意バージョンへの復元
- Agent 変更の一括リバート

### EX-002: スナップショット作成

**目的:**
テスト実行開始時点のテストケースを保持し、実行中にテストケースが編集されても実行結果に影響しない。

**スナップショット対象:**
- テストスイート情報
- テストスイート前提条件
- テストケース一覧
- 各テストケースの前提条件・手順・期待値

### レビューコメント（RC-001〜RC-005）

**コメント対象:**
- テストスイート
- テストケース

**コメント構造:**
```
ReviewComment
├── id
├── targetType (SUITE / CASE)
├── targetId
├── authorId (人 or Agent)
├── content
├── status (OPEN / RESOLVED)
├── createdAt
└── replies[]
    └── ReviewCommentReply
        ├── id
        ├── commentId
        ├── authorId
        ├── content
        └── createdAt
```

**ワークフロー:**
1. Agent がテストケースを作成・編集
2. 人がレビューコメントを登録
3. Agent がコメントを確認して修正
4. Agent がコメントに返信（対応内容を説明）
5. 人が確認してコメントを解決済みにする

### Agent 識別（AG-010, AG-011）

**識別方法:**
MCP リクエストに `X-MCP-Client-Id` ヘッダーを付与し、Agent 操作を識別する。

```
POST /mcp HTTP/1.1
Cookie: session=xxx
X-MCP-Client-Id: claude-desktop-v1.2.3
Content-Type: application/json
```

**Agent 操作の記録:**
- 履歴の `changedBy` に Agent 情報を記録
- レビューコメントの `authorType` で `USER` / `AGENT` を区別
- 編集ロックの `lockedByType` で `USER` / `AGENT` を区別

**セッション管理:**
- Agent は 30 秒間隔でハートビートを送信
- ハートビート途絶から 60 秒後にロックを自動解除
- 無操作 30 分でセッション終了

### 同時編集制御（CE-001〜CE-006）

**ロック対象:**
- テストスイート
- テストケース

**ロック構造:**
```
EditLock
├── id
├── targetType (SUITE / CASE)
├── targetId
├── lockedBy (userId or agentId)
├── lockedByType (USER / AGENT)
├── lockedAt
├── lastHeartbeat
└── expiresAt
```

**MCP 接続管理:**
- Coding Agent は Streamable HTTP で MCP サーバーと連携
- 接続維持中は定期的にハートビートを送信
- 接続切断時（ハートビート途絶）に自動でロック解除

**強制ロック解除:**
- 管理者権限を持つユーザーが任意のロックを強制解除可能
- 解除時にロック保持者（Agent/人）に通知

## データベース設計への影響

この機能一覧から、以下のテーブルが必要：

### 新規追加が必要なテーブル

| テーブル | 用途 |
|---------|------|
| `TestSuitePrecondition` | テストスイートの前提条件（複数） |
| `TestCasePrecondition` | テストケースの前提条件（複数） |
| `TestCaseStep` | テストケースの手順（複数） |
| `TestCaseExpectedResult` | テストケースの期待値（複数） |
| `ProjectHistory` | プロジェクトの変更履歴 |
| `TestSuiteHistory` | テストスイートの変更履歴 |
| `TestCaseHistory` | テストケースの変更履歴 |
| `ReviewComment` | レビューコメント |
| `ReviewCommentReply` | レビューコメントへの返信 |
| `ExecutionSnapshot` | 実行時のスナップショット |
| `ExecutionStepResult` | 各手順の実施結果 |
| `ExecutionPreconditionResult` | 各前提条件の実施結果 |
| `ExecutionExpectedResult` | 各期待値の判定結果 |
| `EditLock` | 編集ロック管理 |

### 既存テーブルの変更

| テーブル | 変更内容 |
|---------|---------|
| `Project` | `ownerId` or `organizationId` の排他制約 |
| `TestCase` | `steps`, `expectedResult` カラムを削除（別テーブル化） |
| `Execution` | スナップショットへの参照追加 |
