# インシデント対応手順

最終更新日: 2025年12月

## 1. 概要

本ドキュメントは、Agentestにおけるインシデント発生時の対応手順を定義します。迅速かつ適切な対応により、サービスへの影響を最小限に抑えることを目的とします。

## 2. インシデントの分類

### 2.1 重大度レベル

| レベル | 定義 | 例 |
|--------|------|-----|
| **Critical (P1)** | サービス全体停止、データ漏洩 | 全ユーザーがアクセス不可、セキュリティ侵害 |
| **High (P2)** | 主要機能の障害、広範囲の影響 | 認証不可、テスト実行失敗、50%以上に影響 |
| **Medium (P3)** | 一部機能の問題、限定的影響 | 特定機能のエラー、パフォーマンス低下 |
| **Low (P4)** | 軽微な問題、回避策あり | UI崩れ、非重要機能の問題 |

### 2.2 インシデントカテゴリ

| カテゴリ | 説明 |
|---------|------|
| Availability | サービス停止、アクセス不可 |
| Performance | レスポンス遅延、タイムアウト |
| Security | 不正アクセス、脆弱性、データ漏洩 |
| Data | データ損失、整合性問題 |
| Integration | 外部サービス連携障害 |

## 4. フェーズ別対応

### 4.1 検知

**アラートソース:**
- 監視システム (Datadog, CloudWatch)
- ログ分析 (異常パターン検知)
- ユーザー報告 (サポート経由)
- セキュリティスキャン

**初動対応:**
1. アラート確認、状況把握
2. ステータスページ更新（調査中）
3. 対応チャンネル作成 (#incident-YYYYMMDD-NN)

### 4.2 トリアージ

**判定項目:**

| 質問 | 判断基準 |
|------|---------|
| 影響ユーザー数 | 全員 → P1、多数 → P2、限定 → P3+ |
| 収益への影響 | 直接影響 → P1-2 |
| データへの影響 | 損失/漏洩 → P1 |
| 回避策の有無 | なし → 重大度上昇 |

**エスカレーション:**

| 重大度 | エスカレーション先 |
|--------|------------------|
| P1 | CTO、全エンジニア |
| P2 | テックリード、オンコール |
| P3 | 担当チーム |
| P4 | 担当者 |

### 4.3 封じ込め

**目的:** 影響の拡大を防ぐ

**対応例:**
- 障害サーバーの切り離し
- 問題機能の無効化
- 不正アクセスのブロック
- トラフィック制限

```bash
# 例: 問題のあるデプロイのロールバック
kubectl rollout undo deployment/api

# 例: 特定IPのブロック
aws waf update-ip-set --name blocked-ips --updates ...
```

### 4.4 根絶

**目的:** 根本原因の特定と除去

**調査手順:**
1. ログ収集・分析
2. タイムライン作成
3. 根本原因の特定
4. 修正コードの作成・レビュー
5. テスト環境での検証

### 4.5 復旧

**目的:** サービスの正常化

**復旧手順:**
1. 修正のデプロイ
2. 段階的なトラフィック復帰
3. 監視強化
4. 正常性確認

**確認項目:**
- [ ] 主要機能の動作確認
- [ ] エラーレートの低下
- [ ] レスポンスタイムの正常化
- [ ] 監視アラートのクリア

### 4.6 事後分析

**ポストモーテム:**

| 項目 | 内容 |
|------|------|
| サマリ | 何が起きたか（1-2文） |
| 影響 | 影響範囲、期間、ユーザー数 |
| タイムライン | 時系列での経緯 |
| 根本原因 | 直接原因と根本原因 |
| 対応内容 | 実施した対応 |
| 再発防止策 | アクションアイテム |
| 教訓 | 学んだこと |

**期限:**
- P1: 48時間以内
- P2: 1週間以内
- P3+: 2週間以内

## 5. コミュニケーション

### 5.1 内部コミュニケーション

| ツール | 用途 |
|--------|------|
| Slack #incident-* | リアルタイム対応 |
| Notion | ポストモーテム記録 |
| PagerDuty | オンコール呼び出し |

### 5.2 外部コミュニケーション

**ステータスページ更新:**

| ステータス | タイミング |
|-----------|-----------|
| 調査中 | 検知後15分以内 |
| 特定済み | 原因特定後 |
| 対応中 | 修正作業中 |
| 監視中 | デプロイ後 |
| 解決済み | 正常確認後 |

**ユーザー通知テンプレート:**

```markdown
## [調査中] サービス障害のお知らせ

現在、一部のユーザーで [問題の概要] が発生しています。
原因を調査中です。ご不便をおかけして申し訳ございません。

- 発生時刻: YYYY-MM-DD HH:MM (JST)
- 影響範囲: [影響を受ける機能]
- 次回更新: 30分後を予定
```

### 5.3 顧客への報告

| 重大度 | 報告方法 |
|--------|---------|
| P1 | 個別メール + ステータスページ |
| P2 | ステータスページ + 週次レポート |
| P3+ | 月次レポート |

## 6. オンコール体制

### 6.1 ローテーション

- **プライマリ**: 週単位でローテーション
- **セカンダリ**: プライマリのバックアップ
- **エスカレーション**: テックリード

### 6.2 応答時間

| 重大度 | 応答時間 |
|--------|---------|
| P1 | 15分以内 |
| P2 | 30分以内 |
| P3+ | 営業時間内 |

### 6.3 ツール

- **PagerDuty**: アラート管理、呼び出し
- **Runbook**: 対応手順書
- **VPN**: リモートアクセス

## 7. 定期訓練

### 7.1 障害訓練

| 種別 | 頻度 | 目的 |
|------|------|------|
| 机上訓練 | 四半期 | 手順確認 |
| 実践訓練 | 半年 | 実動確認 |
| Chaos Engineering | 月次 | 耐障害性検証 |

### 7.2 訓練シナリオ例

- データベース障害
- ネットワーク分断
- セキュリティ侵害
- 大規模トラフィック

## 8. ツール・リソース

### 8.1 監視ダッシュボード

- Grafana: メトリクス可視化
- Datadog: APM、ログ
- CloudWatch: インフラ監視

### 8.2 対応ツール

```bash
# 共通コマンド集

# ログ確認
kubectl logs -f deployment/api

# Pod再起動
kubectl rollout restart deployment/api

# DB接続
docker compose exec db psql -U agentest

# Redis確認
docker compose exec redis redis-cli
```
