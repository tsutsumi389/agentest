# ADR-0001: pnpm の採用

## ステータス

採用

## コンテキスト

Node.js プロジェクトのパッケージマネージャーを選定する必要がある。選択肢は以下の通り：

- npm（Node.js 標準）
- yarn（Facebook 製）
- pnpm（Performant npm）

本プロジェクトはモノレポ構成を予定しており、複数のパッケージを効率的に管理する必要がある。

## 決定

**pnpm を採用する。**

### 選定理由

1. **ディスク効率**
   - シンボリックリンクによる重複排除
   - 同じパッケージを複数プロジェクトで共有

2. **インストール速度**
   - npm/yarn より高速
   - CI/CD の時間短縮

3. **モノレポサポート**
   - `pnpm-workspace.yaml` によるワークスペース管理
   - `--filter` オプションで特定パッケージを操作
   - `catalogs` 機能でバージョン一元管理

4. **厳格な依存関係**
   - 明示的に宣言していないパッケージは使用不可
   - 依存関係の問題を早期発見

## 結果

### メリット

- ディスク使用量の削減
- CI/CD の高速化
- 依存関係の厳格な管理
- モノレポとの相性が良い

### デメリット

- npm/yarn と異なるコマンド体系（学習コスト）
- 一部の古いパッケージで互換性問題の可能性
- node_modules の構造が異なるため、デバッグ時に戸惑う可能性

### 緩和策

- チーム向けに pnpm チートシートを用意
- 互換性問題が発生したら `.npmrc` で `shamefully-hoist=true` を検討
