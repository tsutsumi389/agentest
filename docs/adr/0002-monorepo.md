# ADR-0002: モノレポ構成の採用

## ステータス

採用

## コンテキスト

複数のアプリケーション（Web, Admin, API, WebSocket, MCP Server）と共有パッケージを管理する必要がある。

リポジトリ構成の選択肢：

1. **ポリレポ** - 各アプリを別リポジトリで管理
2. **モノレポ** - 全てを1つのリポジトリで管理

## 決定

**モノレポ構成を採用する。**

ツール構成：
- pnpm workspaces（パッケージ管理）
- Turborepo（ビルドオーケストレーション）

### ディレクトリ構成

```
agentest/
├── apps/           # アプリケーション
│   ├── web/
│   ├── admin/
│   ├── api/
│   ├── ws/
│   └── mcp-server/
├── packages/       # 共有パッケージ
│   ├── shared/
│   ├── db/
│   ├── auth/
│   └── ...
└── docs/           # ドキュメント
```

## 結果

### メリット

- **コード共有が容易**
  - `packages/shared` を全アプリで利用
  - 型定義の一元管理

- **アトミックな変更**
  - API と フロントエンドを同時に変更・コミット
  - 整合性の担保

- **統一された開発体験**
  - 同じ lint/format ルール
  - 同じ CI/CD パイプライン

- **依存関係の可視化**
  - パッケージ間の依存が明確
  - 影響範囲の把握が容易

### デメリット

- **リポジトリサイズの増大**
  - clone に時間がかかる可能性
  - → sparse checkout で緩和可能

- **CI/CD の複雑化**
  - 変更のあったパッケージのみビルドする仕組みが必要
  - → Turborepo のキャッシュで解決

- **権限管理の難しさ**
  - 全員が全コードにアクセス可能
  - → CODEOWNERS で緩和

### 緩和策

- Turborepo の Remote Cache を活用
- CODEOWNERS でコードオーナーを定義
- 必要に応じて sparse checkout を案内
