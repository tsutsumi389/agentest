# ADR-0004: MCP セッション再起動耐性

## ステータス

採用

## コンテキスト

MCP サーバー（`apps/mcp-server`）のセッションは `StreamableHTTPServerTransport` インスタンスとしてインメモリ `Map` に保持されている。SDK の制約上、このトランスポートオブジェクトはシリアライズ不可能であり、以下のシナリオでセッションが消失する問題があった。

**開発環境:**
- `tsx watch` によるホットリロード（コード変更の都度プロセス再起動）
- Docker コンテナの再起動

**本番環境（Cloud Run）:**
- オートスケールによるインスタンス増減
- リクエストが別インスタンスにルーティングされる
- スケールインによるインスタンス停止
- コールドスタート時のセッション不在

セッション消失時、クライアント（Claude Code 等）には `-32600` エラーのみが返され、障害原因の判別や自動復旧が困難だった。

検討した選択肢：

1. **セッションの完全永続化** - Redis/DB にトランスポートをシリアライズして保存
2. **Redis によるセッションルーティング情報の管理** - メタデータのみ Redis に保存し障害原因を判定
3. **スティッキーセッション単独** - ロードバランサーのセッションアフィニティのみに依存

## 決定

**Redis によるセッションルーティング情報の管理（選択肢 2）を採用する。**

トランスポート自体はシリアライズ不可能なため永続化できないが、セッションがどのインスタンスに存在するかのルーティング情報を Redis に保存し、セッション消失時の障害原因判定に使用する。

### セッション 3 層モデル

| 層 | 保存先 | データ | 寿命 |
|---|--------|--------|------|
| MCPトランスポート | インメモリ Map | `StreamableHTTPServerTransport` | プロセス生存中 |
| セッションルーティング | Redis | セッション→インスタンスIDマッピング | TTL 180秒 |
| AgentSession | PostgreSQL | ステータス, ハートビート, プロジェクト紐付け | 永続 |

### ID 体系

| ID | 生成方法 | 寿命 | 用途 |
|---|---------|------|------|
| マシンID | `os.hostname()` / `HOSTNAME` 環境変数 | コンテナ/マシンの生存中 | 同一マシン上での再起動検知 |
| インスタンスID | `randomUUID()` | プロセス生存中 | プロセス単位の一意識別 |

### Redis キー設計

| キー | TTL | 内容 |
|------|-----|------|
| `mcp:instance:{instanceId}` | 120秒 | インスタンス情報（machineId, startedAt） |
| `mcp:session:{sessionId}` | 180秒 | セッションメタデータ（userId, instanceId, machineId） |

### 障害原因の 4 パターン判定

| 原因 | `data.reason` | 判定条件 |
|------|--------------|----------|
| セッション不明 | `session_not_found` | Redis にもセッション情報なし |
| プロセス再起動 | `server_restarted` | Redis にあり & 同じマシンID |
| 別インスタンスが保持 | `wrong_instance` | Redis にあり & 別マシンID & インスタンス生存 |
| インスタンス停止 | `instance_terminated` | Redis にあり & 別マシンID & インスタンス停止 |

全エラーレスポンスに `data.reinitialize: true` を付与し、クライアントの自動復旧の足がかりとする。

## 結果

### メリット

- **障害原因の可視化**
  - 4 パターンの明確なエラー理由により、開発者・運用者がセッション消失の原因を即座に判別可能

- **水平スケール対応**
  - Redis によるセッションルーティング情報でマルチインスタンス環境に対応
  - Cloud Run セッションアフィニティとの組み合わせで最適なルーティング

- **自動復旧への道筋**
  - `data.reinitialize: true` フィールドにより、将来のクライアント自動再接続実装が容易

- **Redis 未設定時の安全なフォールバック**
  - 全 Redis 操作に null チェックを実装し、Redis 未設定環境では従来通りの動作を維持

### デメリット

- **Redis 依存の追加**
  - セッションルーティング機能には Redis が必要（ただしオプショナル）

- **エラーレスポンスの複雑化**
  - `error.data` フィールドの追加（JSON-RPC オプショナルフィールドのため後方互換性は維持）
